# 代码规范

## 目标

实现代码时优先保证：

- 顺着执行路径就能读懂。
- 关键输入、输出、状态变化和副作用清晰可见。
- 概念数量少，不为简单逻辑新增名字。
- 配置、阈值和边界行为显式。
- 同一个业务事实尽量只有一份真相源。

如果一段代码看起来更“抽象”或更“统一”，但阅读路径更绕、隐藏约定更多、维护者更难判断行为边界，那通常不是好实现。

## 核心原则

### 1. 主流程优先

- 按真实执行顺序组织代码。
- 关键判断和对应副作用尽量放在一起。
- 短小、直白、单调用点的逻辑优先内联。
- 不要让主流程只剩一串看不出业务含义的 helper 调用。

### 2. 少加概念

- 新增函数、类型、字段、模块前，先确认它是否表达新的稳定语义。
- 能用已有领域概念表达，就不要新造别名。
- 局部临时数据不要包装成长期概念。
- 简单流程不要提前设计 strategy、provider、registry 等扩展框架。

### 3. 单一真相源

- 一个事实只存一份，其他内容按需派生。
- 不要用两个相近字段互相兜底。
- 不要为了排查方便或展示方便，把同一事实长期双写。
- 必须冗余时，要明确谁是真相源、谁是派生值。

### 4. 显式边界

- 配置、阈值、副作用、外部调用和写入位置要能从代码里直接看出来。
- 不要依赖隐藏环境变量、目录约定或默认行为。
- 内部接口如果有明确调用契约，不要用大量兜底掩盖错误输入。
- 兜底逻辑只在确实服务业务链路时保留，并说明边界。

### 5. 少量重复优于错误抽象

- 少量上下文明确的重复可以接受。
- 不要为了消除几行重复，引入更难理解的参数和中间层。
- 抽象只有在减少真实复杂度、稳定复用或保护边界时才值得做。

### 6. 注释解释约束

- 注释用于说明目的、关键约束、非显然副作用和取舍。
- 不要写机械重复代码表面的注释。
- 如果代码本身已经直白，不需要为了“完整”补注释。

## 常见坏代码形态

### 1. 过度抽象型

表现：

- 为单一调用点新增 helper、manager、adapter、service 或 wrapper。
- 把几行直白判断拆成多个函数。
- 当前只有一种场景，却提前做通用框架。

应该优先：

- 把短小逻辑留在调用处。
- 只有逻辑有独立业务语义、稳定复用，或留在原处明显打断主流程时，才提炼函数。

### 2. 防御性封装型

表现：

- 内部接口参数有明确契约，却写大量合法性判断、默认兜底、catch 后回退。
- 简单解析逻辑被封装成 `readXxx`、`resolveXxx` 这类只服务单点调用的函数。
- 为避免错误暴露，把错误输入悄悄转成默认值。

问题：

- 主流程被打散，真实调用契约变模糊。
- 错误数据被吞掉，后续更难定位问题。
- 代码看似更稳，其实隐藏了边界问题。

应该优先：

- 内部 API 按明确契约直接读取输入。
- 短小解析逻辑直接写在路由或调用处。
- 让不符合契约的错误正常暴露，除非业务明确要求降级。

### 3. 兜底掩盖边界型

表现：

- 大量 fallback、可选字段、宽松兼容逻辑。
- 上游语义不清，下游不断补默认值。
- 写入时维护多份相近事实，读取时再猜哪份有效。

应该优先：

- 回到负责这段语义的位置明确输入输出。
- 修正契约，而不是继续加兜底。

### 4. 改动扩散型

表现：

- 为一个局部问题顺手改无关模块、命名、目录或类型。
- 借修 bug 做大范围重构。
- 修改范围超过当前需求，但没有说明必要性。

应该优先：

- 只改当前问题必要触达的文件和逻辑。
- 确实需要重构时，先说明原因、范围和收益。

### 5. 流程断裂型

表现：

- 主流程只剩多个间接调用。
- 条件判断和对应动作距离很远。
- 状态变化、外部调用和错误处理藏在深层函数里。

应该优先：

- 让关键判断、状态变化和副作用在主流程中可见。
- 按真实执行顺序展开，不为“结构整齐”牺牲阅读路径。

### 6. 类型表演型

表现：

- 为局部临时数据新增复杂类型、泛型或映射结构。
- 类型体系比业务流程更复杂。
- 为了复用类型，调用处失去具体业务语义。

应该优先：

- 使用贴近业务的数据结构。
- 只在类型能表达稳定边界、减少误用或提升阅读体验时新增类型。

### 7. 语义漂移型

表现：

- 字段名、函数名或类型名和真实业务含义不一致。
- 为复用旧概念，把新语义塞进不贴切的结构。
- 同一含义在不同层使用不同名称。

应该优先：

- 先校正命名和职责边界，再调整结构。
- 新增概念前确认它确实代表新的业务事实。

## 自查清单

提交实现前检查：

- 主流程是否顺着执行路径就能读懂？
- 新增函数、类型、字段是否真的提供新语义？
- 是否引入了第二份真相源？
- 配置、阈值和副作用是否显式？
- 抽象是在减少复杂度，还是只是移动复杂度？
- 删除一个中间层后，代码会不会更清楚？
- 有没有用防御性兜底掩盖内部接口契约？
- 注释是否解释关键约束，而不是重复代码表面？

## 一句话原则

让代码更直接、更清楚、更可预测；减少不必要的概念、层级和重复真相源；让阅读者顺着主流程就能理解系统怎么工作。
