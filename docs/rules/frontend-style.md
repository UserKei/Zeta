# 前端样式规则

## 目标

前端样式实现优先保证页面结构清楚、控件语义稳定、滚动区域可预测。布局和视觉调整应服务真实页面关系，不用全局工具类或 magic number 掩盖局部问题。

## 页面布局

- 页面级布局优先使用 `flex` 或 `grid` 表达父子关系。
- 需要占满剩余高度时，由父容器建立高度上下文，子内容区使用 `flex-1 min-h-0`。
- 滚动应放在明确的内容区域，优先使用 `overflow-auto`，或让 `el-scrollbar` 包裹 `flex-1 min-h-0` 的区域。
- 不为单个页面抽全局高度工具类；只有至少 3 个页面稳定复用同一套高度计算时，才考虑抽成公共布局规则。

## 固定高度

- 固定高度只描述组件，不描述页面。
- 固定高度可以用于局部组件自身尺寸，例如图标按钮、普通按钮、输入框、工具栏、表格行高、Tag 高度、上传 dropzone 的最小高度、编辑器初始高度。
- 弹窗内为了防止长内容撑爆，可以使用 `max-h` 限制内容区高度。
- 页面主区域、左右栏、列表区、目录区、预览区不应依赖固定高度分配空间。
- 避免在主内容区写死 `height="560px"`、`height="600px"`、`min-h-[620px]` 这类承担页面布局职责的固定高度。
- 优先由上层 Layout 提供高度上下文；如果短期无法调整父布局，只允许在页面壳层用一个局部 `calc(100vh - ...)` 兜底。
- 禁止把 `calc` 或固定高度下沉到列表、目录、预览区等内部内容。

## Tailwind 与 CSS

- 页面外层布局、间距、边框、背景优先使用 Tailwind。
- 颜色优先使用 Zeta 语义变量，例如 `bg-(--zeta-panel)`、`text-(--zeta-muted)`、`border-(--zeta-line)`。
- 避免在页面里散落硬编码颜色；确实需要新增颜色时，先补充 `--zeta-*` 语义变量。
- 少量复杂状态、第三方组件覆盖或 Markdown 内容样式，可以保留 scoped CSS。
- 不为了“纯 Tailwind”牺牲可读性；模板 class 过长时，可以用语义化容器和局部 CSS 收口。

## Element Plus

- 后台管理页优先使用 Element Plus 的原生组件能力，而不是强行覆盖内部结构。
- 导航、表格、表单、弹窗、确认框、Tag、Message 等优先使用 Element Plus。
- 避免对 `el-button`、`el-menu` 等组件写大量 `!`、`[&>span]`、深层 display 覆盖。
- 只有在组件 API 无法表达必要样式时，才使用 `:deep(...)`，并把范围限制在当前页面或组件内。

## 页面类型

- 管理端页面保持安静、清晰、可扫描，优先保证表格、表单、操作区稳定。
- Chat 页面可以使用更自定义的 Tailwind 布局，减少后台表单感，但仍保持浅色专业风格。
- 上传、分段、聊天这类流程页要让主操作区域吃满可用空间，避免底部大块无意义留白。

## 响应式

- 先保证 375px、768px、1440px 三类宽度下不溢出、不重叠。
- 固定宽度必须有响应式兜底，例如移动端 `w-full`、桌面端再限制 `max-w-*` 或比例宽度。
- 工具栏在窄屏下允许换行，但按钮和输入框不能互相遮挡。
