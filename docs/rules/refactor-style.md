# 重构风格规则

## 目标

重构的目标不是让代码看起来更抽象、更“高级”或更通用，而是让代码更容易理解、更容易维护，并且更贴近真实执行路径。

好的重构应该带来这些结果：

- 阅读者能顺着业务流程自然看懂代码在做什么
- 关键状态、输入输出和副作用清晰可见
- 代码中的概念数量减少，而不是增加
- 修改一个需求时，更容易定位真正需要改动的地方

如果一次重构只是把复杂度从一个地方搬到另一个地方，或者引入更多中间层、中间结构和抽象概念，那通常不是好的重构。

## 核心判断标准

### 1. 可读性优先于抽象上的“优雅”

判断一段代码是否合理，首先看它是不是容易读懂，而不是看它是否足够抽象、是否足够“通用”。

优先选择：

- 顺着执行路径就能看懂的代码
- 变量、函数和类型名称直白表达语义
- 状态变化和副作用发生的位置明确

谨慎对待：

- 需要在多个文件之间来回跳转才能理解的实现
- 为了减少几行重复而引入额外抽象
- 需要先理解“框架内框架”才能读懂的封装

### 2. 显式优于隐式

依赖、条件、状态变化和边界行为应该尽量显式表达，不要依赖隐含约定、默认行为或隐藏在封装内部的规则。

优先选择：

- 显式参数
- 明确的数据结构
- 可以直接从代码中读出的执行条件

避免：

- 看名字猜不出行为的封装
- 把关键逻辑藏在 helper、adapter、manager、wrapper 内部
- 不看上下文就无法判断副作用的函数

### 3. 面向当前问题，而不是假想未来

重构首先服务当前真实需求，而不是为未来可能出现但尚未发生的变化提前铺设复杂结构。

优先选择：

- 能直接解决当前问题的最小设计
- 只抽象已经稳定、已经重复、已经明确有独立语义的部分

避免：

- 为“以后也许会复用”而提前抽象
- 为“以后也许会扩展”而增加额外层级
- 把当前只有一种语义的逻辑包装成通用模型

### 4. 允许少量重复，避免错误抽象

少量、可接受的重复通常比错误的抽象更容易维护。

如果两段逻辑虽然相似，但语义、上下文或变化方向不同，那么保持分别实现，通常比强行统一更清楚。

应当警惕这类情况：

- 为了去掉重复，导致函数参数越来越多
- 为了统一结构，导致调用处失去业务语义
- 为了复用，导致代码阅读路径被拉长

## 推荐做法

### 1. 优先写贴近执行路径的代码

代码应尽量按照真实执行顺序组织，让阅读者可以顺着流程往下看，而不是不断跳出当前上下文。

适合直接内联的逻辑，就不要为了形式上的整洁拆出去。  
如果一段逻辑只有一个调用点，而且本身很短、很直白、没有独立语义，通常应该保留在调用处。

### 2. 函数提炼要有明确语义

提炼函数不是为了缩短代码，而是为了抽出“值得单独命名的一段语义”。

适合提炼函数的情况：

- 这段逻辑本身有独立业务含义
- 提炼后，函数名能明显提升理解速度
- 这段逻辑有多个调用点，而且行为保持一致
- 这段逻辑较长，留在原地会打断主流程理解

不适合提炼函数的情况：

- 只是几行简单赋值或判断
- 提炼后函数名仍然很空泛
- 调用者反而要跳转过去才能知道它做了什么
- 只有一个调用点，且逻辑本身一眼能看懂

### 3. 直接产出结果，减少中间结构

如果目标结果可以直接构造出来，就不要为了“分层整齐”额外引入多余的中间对象、中间状态和中间模型。

优先选择：

- 直接从输入推导输出
- 在必要的位置完成转换
- 尽量减少“先包装再拆开”的数据流

避免：

- 明明只在本地使用，却先定义一层额外结构
- 先变成通用格式，再在下游还原成具体格式
- 为了形式统一而丢失原本清晰的业务语义

### 4. 尽量缩短理解链路

一段代码是否好维护，往往取决于理解它需要经过多少层跳转。

理想状态是：

- 入口清楚
- 主流程清楚
- 分支条件清楚
- 结果落点清楚

如果为了“结构美观”把一条简单链路拆成多层调用，最终让阅读者必须频繁切换上下文，这种重构通常是负收益。

### 5. 让副作用清晰可见

涉及状态更新、写入、删除、外部调用、日志等副作用时，应尽量让副作用发生的位置清晰可见。

优先选择：

- 在主流程中显式调用副作用
- 让副作用与条件判断保持就近关系
- 用命名直白的代码描述真正发生的动作

避免：

- 调一个看似普通的工具函数，却在内部隐式修改状态
- 让副作用穿透多层封装后才发生
- 把关键状态变更藏在默认分支或兜底逻辑里

## 反模式

### 1. 为抽象而抽象

常见表现：

- 代码本来很简单，却额外拆出一层 helper
- 逻辑只有一个调用点，却为了“结构统一”拆模块
- 只是名称变化，却被包装成“通用能力”

### 2. 用统一模型掩盖不同语义

不同业务含义的数据，如果被强行塞进同一种抽象结构，表面上看更统一，实际上会增加理解成本。

当一个统一模型让调用方必须额外记住很多“字段在不同场景下代表不同含义”的规则时，这通常说明抽象方向错了。

### 3. 引入不必要的中间层

常见表现：

- router 只做透传
- manager 只做转发
- wrapper 只改个名字再继续调用
- adapter 只为适配当前唯一调用点而存在

如果一层封装没有新增清晰语义、没有隔离真实复杂性、没有改善调用体验，那么它很可能只是增加理解负担。

### 4. 保险式设计过多

为了“更稳妥”而叠加多层兜底、重复校验、额外状态记录，如果没有明确收益，就会把代码越写越重。

应警惕：

- 已有上游约束，却在下游重复做同类校验
- 内部过程已经能自我保证，却又让外层感知中间态
- 用额外机制弥补本可通过职责边界解决的问题

### 5. 把简单问题做成框架

当一个局部问题的解决方案需要先理解一套小型体系、一组规则或一条多阶段链路时，通常说明设计已经过重。

局部问题优先使用局部解法，不要把简单流程上升成需要长期维护的小框架。

## 什么时候应该删除，而不是继续封装

遇到下面这些情况，优先考虑删除、合并、内联，而不是继续加抽象：

- 一层封装没有独立语义，只是改名或透传
- 一个类型只是把已有数据重新包一层
- 一个函数只有一个调用点，且逻辑本身短小直白
- 一个中间状态只为弥补设计问题而存在
- 一个统一模型让调用方理解成本明显升高
- 一个兼容分支已经不再需要

重构不是不断“补层”，而是不断判断哪些东西其实不该存在。

## 重构检查清单

在提交一段重构前，可以用下面的问题自查：

- 删除这一层后，代码会不会更容易理解？
- 这次提炼函数，是真的抽出了独立语义，还是只是把代码搬走？
- 这个抽象是在减少复杂度，还是在转移复杂度？
- 这个“通用性”是当前真实需要，还是假想未来？
- 这份统一结构是否让语义更清楚，还是更模糊？
- 是否可以接受少量重复，从而换来更短的理解路径？
- 阅读主流程时，状态变化和副作用是否清晰可见？
- 新增的概念、类型、模块和层级，是否真的值得它们带来的认知成本？

## 一句话原则

重构的方向应该是：

让代码更直接，而不是更绕；  
让语义更清楚，而不是更统一；  
让主流程更可读，而不是让抽象层次更多。
